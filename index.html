왜 AlphaForge 인가?

AlphaForge는 단순히 돈을 벌어다 주는 블랙박스 소프트웨어가 아닙니다. 이것은 설계자의 투자 철학과 운용 원칙을 외부 신경계(external nervous system)로 시스템화한 정교한 결정체입니다. 본 시스템은 AI의 창의성보다 재현 가능성(reproducibility)에 집중합니다.

⌘

외부 신경계 모델

AI를 독립적인 의사결정자로 두지 않고, 설계자의 두뇌를 확장한 외부 신경계로 정의합니다. 어제 수익을 냈던 논리가 오늘 LLM의 변덕으로 바뀌어서는 안 됩니다.

⊠

환각의 시스템적 차단

통제되지 않은 AI는 금융 시스템 내에서 치명적인 환각(Hallucination)을 일으켜 자산을 순식간에 소멸시킬 수 있습니다. AHE 5개 레이어와 KFinOnto 온톨로지로 환각을 사전·사중에 차단합니다.

◇

화이트박스 시스템

모든 의사결정에 reason_code와 reason_text가 첨부됩니다. "AI가 거부했습니다"가 아니라 "VaR 3.2%가 한도 3.0%를 초과했습니다"처럼 측정 가능한 metric으로 설명됩니다.

▣

Hash-Chained Audit

모든 의사결정이 PostgreSQL audit_trail에 SHA-256 hash chain으로 기록되며, 매주 OpenTimestamps로 비트코인 블록체인에 앵커링되어 사후 변조가 암호학적으로 불가능합니다.

◐

가시성 ≠ 가독성

가시성은 시스템의 책임, 가독성은 운용자의 책임. 평시 90%엔 핵심 5개 지표만, 깊이 볼 때 11개 노드 흐름 즉시 펼침. 보잉 787 글래스 콕핏을 모델로 합니다.

◈

오케스트라 모델

LangGraph Supervisor(마에스트로)가 10명의 Specialist Agent를 지휘. 단원이 자기 악기만 연주하듯, 각 에이전트는 tools.profile에 명시된 도구만 사용 가능. 권한 분리(Privilege Separation)가 코드 레벨에서 강제됩니다.

Design Principles

절대 위반하지 않는 6대 설계 원칙

Brian의 45년 엔지니어링 경험과 AEGIS · LLMOps · wdlab-CLI · LOAN4U · ARGUS · NexusAI · FileForge에서 검증된 운영 노하우에서 도출된 6가지 원칙. 이 원칙은 어떤 경우에도 우회하지 않습니다.

PRINCIPLE 01

기존 API 계층 비변경

vLLM, Ray Serve, Wazuh, Kafka, OpenSearch 등 기존 인프라는 그대로 둡니다. AlphaForge가 추가되어도 기존 워크로드의 안정성에 영향 0.

PRINCIPLE 02

추가형 얇은 라우팅 셸

OpenClaw가 채널·MCP·도구 라우팅만 담당. 비즈니스 로직은 vLLM 뒷단의 순수 API 계층에서. OpenClaw는 트레이딩 엔진이 아니라 게이트웨이.

PRINCIPLE 03

80% 이상 인프라 재사용

AHE 적용 시 약 87% 재사용률. 신규 컴포넌트는 Broker MCP, Strategy SKILLs, RiskGuard 모듈로 한정.

PRINCIPLE 04

멀티 에이전트 하네스 엔지니어링

LangGraph Supervisor 위에 10명의 Specialist를 배치하는 패턴은 ARGUS와 동형(isomorphic). 동일 운영 패턴이므로 학습 비용 0.

PRINCIPLE 05

오픈소스 우선

상용 SaaS 의존을 최소화. 외부 LLM 호출은 prefix block(gpt-, claude-, azure/)으로 차단. 단, 브로커 API는 불가피한 외부 의존.

PRINCIPLE 06

5단계 게이팅

Backtest → Paper → Shadow → Limited Live → Full Live. 단계 간 자동 승급은 절대 금지. 사람의 명시적 수동 승인을 통해서만 진입 가능.

White-box Promise

화이트박스의 7가지 구조적 증거

AlphaForge가 화이트박스 시스템임은 다음 7가지 구조적 증거로 입증됩니다. 이 증거들은 운용자가 시스템을 신뢰하기 위해 필요한 모든 검증 가능한 정보를 제공합니다.

①

소스 코드 공개

모든 코드가 Brian의 GitHub 조직(wdlab1958) 안에 있습니다. 외부 의존 코드는 모두 라이선스가 명시된 오픈소스입니다.

②

명시적 데이터 구조

Schema Layer가 모든 노드 간 전이를 Pydantic 모델로 강제합니다. OrderProposal·Signal·RiskDecision·DecisionToken·AuditEntry.

③

강제 감사 기록

Audit Layer가 모든 의사결정을 hash-chained PostgreSQL audit_trail + OpenSearch alphaforge-audit-*에 기록.

④

암호 서명된 권한

Decision Token이 모든 권한 행사를 Ed25519로 서명. 토큰 없는 주문은 broker MCP 호출 자체가 차단됩니다.

⑤

결정의 재현성

Determinism Layer가 LLM 호출을 캐시(sha256(prompt+input+model+seed+temp)). Backtest와 Live가 동일한 의사결정 경로를 따릅니다.

⑥

추적 가능한 진화

Evolution Layer가 GEPA 진화를 git처럼 버전 추적. Bonferroni correction으로 통계적 유의성 검증, 회귀 시 자동 롤백.

⑦

명시적 사유 첨부

모든 의사결정에 reason_code(예: VAR_EXCEEDED)와 reason_text가 첨부됩니다. 측정 가능한 metric으로 설명.

기술의 효율성을 믿되, 그 효율성이 인간의 통제를 벗어나는 순간을 잊지 마십시오. AlphaForge는 철저하게 계산된 공학적 안전 위에서 춤추는 지적인 여정입니다.
— Brian Lee, Research Director, WDLAB@2023-2026 Co., Ltd.
Agent Harness Engineering

야생마에 마구를 채우는 AHE 5개 메타 레이어

AHE(Agent Harness Engineering)는 야생마에게 마구(Harness)를 채워 마차를 끌게 하듯, AI의 추론 능력을 강력한 코드 게이트 안에 가두는 기술입니다. 11개 에이전트 노드 위에 횡단(cross-cutting)으로 적용됩니다.

L1

Schema Layer

Pydantic 강제10+ Modelsfield_validator

모든 노드 간 state 전이를 Pydantic 모델로 강제. OrderProposal·Signal·RiskDecision·DecisionToken·AuditEntry·AlphaForgeState 등 10+ 모델. 입력 자체가 변조 불가능.

L2

Audit Layer

Hash ChainSHA-256OpenTimestampsWORM

Hash-chained PostgreSQL audit_trail + OpenSearch 인덱스. cycle_id 기반 11개 노드 흐름 재구성 가능. SHA-256 chain 무결성 깨지면 자동 정지. 매주 OpenTimestamps 비트코인 앵커링.

L3

Policy Layer

3-Gatetools.profileproviderPolicyEd25519

3-gate 권한 분리 — tools.profile(에이전트별 도구) + providerPolicy(외부 LLM 차단) + Decision Token Ed25519 10초 TTL. ExecutionAgent의 place_order는 토큰 없으면 OpenClaw 하네스에서 차단.

L4

Determinism Layer

LLM CacheBacktest = Liveseed + temp

LLM 호출 결과 캐시. 캐시 키 = sha256(prompt_template_version + input_hash + model + seed + temperature). 동일 입력에 동일 출력을 보장하여 Backtest와 Live가 동일 결정 경로.

L5

Evolution Layer

GEPA 0~2세대SkillForgeBonferroniAuto-RollbackBayesian Optimization

GEPA(Genetic-Evolution Prompt Adaptation) 자동 진화 + SkillForge 자동 후보 생성. Bonferroni correction으로 통계적 유의성 검증. 3세대 이상은 overfit 위험으로 보수적 제한. 진화 대상은 프롬프트 텍스트·retrieval top-k·temperature 같은 parameter만이며, 매매 로직·RiskGuard 한도·11개 노드 구조는 절대 진화 대상이 아닙니다.

Multi-Agent Orchestra

오케스트라의 11명의 단원

LangGraph Supervisor(마에스트로) 1명 + 10명의 Specialist Agent. 정보 흐름의 단계에 따라 4그룹으로 분류됩니다. 엄격한 품질 검사관인 RiskGuard의 승인 없이는 어떤 신호도 시장에 도달할 수 없습니다.

Pre-Trade Validation

RiskGuard Stage 0 → 6단계 검증 파이프라인

모든 매매 주문은 Stage 0 (KFinOnto ontology pre-check, Phase 67) 이후 다음 6단계 관문을 순차적으로 통과해야 합니다 — 실효적 7단 평가. 한 단계라도 실패하면 즉시 DENY되며, Fail-Closed 모드로 진입합니다. 평가 시간 < 10ms를 유지하여 실시간 거래에 영향을 주지 않습니다.

Heartbeat & Self-Halt

RiskGuard 자체가 죽으면 시스템 전체가 fail-closed로 진입. Redis 30초 TTL heartbeat key. RiskGuard 응답 없으면 ExecutionAgent도 fail-closed. 재기동 시 마지막 상태를 PostgreSQL에서 복구.

Cryptographic Authorization

위조 불가능한 Decision Token

RiskGuard의 6단계 검증을 통과한 주문에는 결정 토큰(Decision Token)이라는 디지털 서명이 부여됩니다. 이는 공장의 품질 검사관이 찍는 최종 합격 도장과 같습니다. StrategyAgent → RiskGuardAgent(서명) → ExecutionAgent의 권한 분리(RBAC)를 암호학적으로 강제합니다.

Ed25519 · TTL 10s

디지털 도장 비유

"도장이 있어야만 출하 가능"의 정확한 구현.

  • 알고리즘Ed25519 (Edwards-curve, Curve25519), 64 bytes signature
  • 키 관리alphaforge-keygen CLI · keys/ 폴더 mode 0600
  • TTL10초 (Replay attack 차단)
  • 발행자RiskGuard 단독. StrategyAgent는 발행 권한 없음
  • 검증 위치broker MCP gateway에서 서명 재검증
  • 멱등 키audit_id = idempotency_key. Redis 24h 캐시
  • Claim 커버리지symbol·side·qty·order_type·limit_price 전체

5-Stage Operation Gating

자전거 비유로 설명되는 5단계 운용 게이트

처음부터 두 발 자전거를 타는 사람은 없습니다. 보조바퀴(Stage 1~2)로 균형을 잡고, 평지 천천히(Stage 3) 자신감을 쌓고, 짧은 거리(Stage 4)부터 시작하여 마지막에 자유롭게(Stage 5) 달립니다. 단계 간 자동 승급은 코드와 인프라 양쪽에 다중으로 차단되어 있습니다.

Emergency & Auto-Recovery

비상 운용과 자동 회복

본 시스템은 정지 상태에서 손실이 더 커지지 않도록 설계되어 있습니다. 기술이 통제 불능에 빠졌다고 느낄 때, 두 번 생각하지 말고 /halt를 누르세요.

■

① 대시보드의 빨간 [EMERGENCY HALT] 버튼

③ 자동 트리거 — 일일 손실 한도 도달 · heartbeat 끊김 · hash chain 무결성 깨짐 · 양 broker 모두 실패

⊕

비상 정지 시 8단계 동작

① halt 신호 → 모든 cycle
② 진행 중 cycle은 현재 노드 완료 후 즉시 종료
③ 미체결 주문 모두 자동 취소
④ Live: 보유 포지션 시장가 청산 (자본 보호 우선)
⑤ Paper: 포지션 보존 (학습 목적)
⑥ system_state DB에 halted=true 기록
⑦ Telegram "🚨 SYSTEM HALTED" 알림
⑧ /resume 전까지 신규 cycle 0

Operating Doctrine

운용 3대 핵심 수칙

이 3대 수칙은 단순한 가이드라인이 아니라, AlphaForge 67 Phase의 모든 결정이 향하는 목적지입니다.

01

안전 우선

어떤 분석가 노드의 신호가 아무리 강해도, RiskGuard가 한도 위반을 감지하면 무조건 거부됩니다. 이 우선순위를 뒤집으려는 모든 시도(예: "이번 한 번만 한도 초과 허용")는 시스템의 안전 모델을 무너뜨립니다.

02

단계 엄수

Backtest 4주 + Paper 4주 + Shadow 2주 + Limited Live 4주 = 총 14주. 이는 기술적 검증 시간이 아니라 운용자의 직관 형성 시간이기도 합니다. 이 시간 동안 운용자는 시스템 패턴, 평소 latency, 거부 사유 분포를 체화합니다.

03

인간 개입

기술이 통제 불능에 빠졌다고 느낄 때, 두 번 생각하지 말고 /halt를 누르세요. 시스템은 정지 상태에서 손실이 더 커지지 않도록 설계되어 있습니다. 정지 후 천천히 원인을 점검하면 됩니다.

Roadmap · Vision

AlphaForge의 최종 목표

단기는 운용 안정성, 중기는 단계적 자본 확장, 장기는 투자자문 플랫폼으로의 전환. 궁극적으로는 AI 자동매매의 표준 안전 모델로 자리잡는 것이 목표입니다.
